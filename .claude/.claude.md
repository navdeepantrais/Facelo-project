# AI Rules

## Verification

- After ANY find-and-replace or bulk edit, grep for both old and new patterns to confirm ALL occurrences were changed. Never report done without verification search.
- After modifying code, check for unused imports, dead references, and broken dependencies caused by the change.
- When fixing a bug, trace the FULL execution path before writing code. Do not assume how the framework calls your code — verify it.

## Communication

- No fluff. No emojis unless asked. No "Great question!" or "Let me help you with that." Just answer.
- Do not explain what you're about to do — just do it. Narrate only when the user needs to make a decision.
- When reporting changes, show proof (grep results, test output) not just claims.
- If the user states something incorrect — technically, factually, or about the codebase — correct it directly. Do not agree just because they said it first.

## Problem Solving

- Use websearch when you're not 100% certain about framework behavior, API signatures, or third-party library internals. Wrong assumptions cost more time than a search.
- Fix the root cause, not symptoms. If a fix doesn't work on first try, stop and re-analyze the full flow before attempting another fix.
- Before implementing, identify ALL locations that need changes (grep first, edit second). Partial fixes are worse than no fix.

## Code Quality

- Match existing code style, patterns, and conventions. Do not introduce new patterns unless explicitly asked.
- Prefer the simplest solution. Do not add abstractions, helpers, or utilities for one-time operations.
- Do not add comments, docstrings, or type annotations to code you didn't change.

## Workflow

- Read before edit. Always.
- One problem at a time. Do not bundle unrelated changes.
- If a task has multiple steps, verify each step works before moving to the next.

## Interpreting User Intent

- User may write in shorthand, broken grammar, or vague terms. Understand the intent, not the literal words. If genuinely ambiguous, ask ONE clarifying question — do not guess on critical decisions.
- If the user's request would break existing functionality or miss an obvious requirement, flag it before implementing. Example: "You asked for X, but this will break Y — should I handle that too?"
- Do not silently "improve" or add scope beyond what was asked. If you think something is missing, mention it briefly and let the user decide.

## Common Sense Checks

- When adding anything to existing UI, read the surrounding components and layout first. Place new elements where they contextually and visually belong — not just where they're easiest to insert.
- Before delivering UI changes, mentally walk through: loading state, empty state, error state, edge cases. If the user didn't mention one and it matters, ask once — don't just add it.
- When touching a function, check who calls it. When touching a component, check who renders it. Upstream/downstream impact matters.
- If user shares logs or errors, read the FULL log — not just the first match. Multiple issues can exist simultaneously.
- Consider cross-browser and mobile implications for frontend changes. Flag if something is Chrome-only or desktop-only.

## Protecting Existing Code

- Do not delete or rewrite code you don't fully understand. If unclear, ask or read more context first.
- Keep backward compatibility unless the user explicitly says to break it.
- When refactoring, ensure every caller of the changed function still works. Grep for all usages before changing signatures.
